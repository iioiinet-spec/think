트라이 Trie

접두사를 공유하는 트리로 사전을 담다 — 길이 L의 단어를 O(L)에 찾는다

C 시각화: 노드 생성 · isEnd · 탐색 vs 접두사 · 자동완성

사전을 어떻게 담을까 — 접두사의 발견

단어 수십만 개를 담고 "이 단어가 사전에 있나?", "이 접두사로 시작하는 단어는?"을 빠르게 답해야 합니다. 어떻게 할까요?

방법단어 검색접두사 검색
배열 + 순회O(N·L)O(N·L)
정렬 + 이진 탐색O(L log N)O(L log N)
해시셋O(L)불가능!
트라이O(L)O(L)
💡 해시셋의 결정적 약점 — 단어 검색은 O(L)로 빠르지만, "ca로 시작하는 단어"를 못 찾습니다. 해시는 문자열을 통째로 뭉개버려 접두사 정보가 사라지거든요. 트라이는 글자를 하나씩 트리로 펼쳐 접두사를 자연스럽게 다룹니다.
트라이(Trie)의 아이디어 — 단어를 글자 하나 = 간선 하나로 트리에 새깁니다. 루트에서 어떤 노드까지의 경로가 곧 접두사예요. cat과 car은 ca까지 같은 길을 쓰다가 갈라집니다. 이름은 retrieval(검색)에서 왔어요.

이 강의의 예제 사전

단어 6개 — cat, car, card, care, do, dog. 접두사를 많이 공유하도록 골랐습니다.

완성된 트라이
노드 10개 (root 포함) 글자 총합 19개 → 공유 없이 만들면 20개 노드 필요 → 접두사 공유로 절반으로 줄었다! 초록 이중원 = isEnd (여기까지가 단어)
💡 두 가지를 눈여겨보세요. ① ca 노드는 단어가 아닙니다(초록 아님) — 지나가는 길일 뿐이에요. ② do 노드는 단어이면서 동시에 dog로 가는 길이기도 합니다. 이 둘을 구분하는 게 isEnd 플래그예요.

노드의 구조

C — 트라이 노드
#define ALPHA 26

typedef struct Node {
    struct Node* child[ALPHA];
    int isEnd;   /* 여기까지가 단어? */
} Node;

Node* newNode(void){
    return calloc(1, sizeof(Node));
    /* calloc → child 전부 NULL,
       isEnd = 0 으로 초기화 */
}
두 정보의 역할
// child[c] : 글자 c로 가는 자식
//   'a'→child[0], 'b'→child[1]...
//   NULL이면 그 글자로 못 감

// isEnd : 이 노드에서 끝나는
//         단어가 있는가?

// 노드 자체는 글자를 저장 안 함!
// 글자는 "간선"(배열 인덱스)에
// 담겨 있음
글자는 노드가 아니라 간선에 있다 — 헷갈리기 쉬운 부분입니다. 노드는 글자를 저장하지 않아요. child[2]가 가리키는 자식은 "글자 c로 가는 길"이라는 뜻이고, 글자 정보는 배열 인덱스 자체에 들어 있습니다. 그래서 노드에는 isEnd와 자식 포인터만 있으면 돼요.

삽입 — 없으면 만들며 내려간다

단어를 넣는 건 단순합니다. 루트에서 시작해 글자를 하나씩 따라 내려가요. 가는 길이 없으면 만들고, 있으면 그냥 재사용합니다. 마지막 노드에 isEnd = 1을 찍으면 끝.

C — 삽입
void insert(const char* s){
    Node* cur = root;
    for (int i=0; s[i]; i++){
        int c = s[i] - 'a';
        if (!cur->child[c])
            cur->child[c] = newNode();  /* 없으면 생성 */
        cur = cur->child[c];         /* 한 칸 내려감 */
    }
    cur->isEnd = 1;   /* 여기까지가 단어! */
}

// → 이게 트라이가 아끼는 방식
💡 복잡도는 O(L)입니다(L = 단어 길이). 사전에 단어가 몇 개든 상관없어요. 내가 넣는 단어의 길이만큼만 내려가면 되니까요. 이게 트라이의 매력입니다.

탐색과 접두사 — isEnd 하나가 가르는 차이

이 편에서 가장 중요한 구분입니다. 두 함수는 거의 똑같이 생겼는데 마지막 한 줄이 다릅니다.

C — search (단어가 있나?)
int search(const char* s){
    Node* cur = root;
    for (int i=0; s[i]; i++){
        int c = s[i] - 'a';
        if (!cur->child[c])
            return 0;    /* 길이 끊김 */
        cur = cur->child[c];
    }
    return cur->isEnd;  /* ← 이 한 줄! */
}
C — startsWith (접두사가 있나?)
int startsWith(const char* s){
    Node* cur = root;
    for (int i=0; s[i]; i++){
        int c = s[i] - 'a';
        if (!cur->child[c])
            return 0;    /* 길이 끊김 */
        cur = cur->child[c];
    }
    return 1;          /* ← 도착만 하면 OK */
}
search("ca") = false   하지만   startsWith("ca") = true
왜 다른가 — ca까지 가는 길은 있습니다(cat, car이 지나가니까). 그래서 startsWith는 true. 하지만 ca라는 단어를 넣은 적은 없으니 isEnd = 0이고, search는 false예요. "길이 있다"와 "단어다"는 다릅니다.

자동완성 — 트라이가 빛나는 순간

검색창에 ca를 치면 car, card, care, cat이 뜨는 그 기능입니다. 트라이로는 놀랍도록 자연스러워요.

두 단계면 끝 — ① 접두사를 따라 내려가 그 지점의 노드를 찾고(O(L)), ② 그 노드 아래를 전부 DFS하며 isEnd인 곳마다 단어를 수집합니다. 접두사로 시작하는 단어들은 모두 그 서브트리 안에 있으니까요!

C — 수집 (DFS)
char buf[64];
void collect(Node* p, int depth){
    if (p->isEnd){
        buf[depth] = '\0';
        printf(" %s", buf);   /* 단어 발견! */
    }
    for (int c=0; c<ALPHA; c++)
        if (p->child[c]){
            buf[depth] = 'a'+c;
            collect(p->child[c], depth+1);
        }
}
C — 자동완성
void autocomplete(const char* s){
    Node* cur = root;
    /* ① 접두사까지 내려가기 */
    for (int i=0; s[i]; i++){
        int c = s[i]-'a';
        if (!cur->child[c]) return;
        cur = cur->child[c];
    }
    strcpy(buf, s);
    /* ② 그 아래 전부 수집 */
    collect(cur, strlen(s));
}
💡 a부터 z 순서로 DFS하면 결과가 자동으로 사전순입니다. 따로 정렬할 필요가 없어요. 트라이 자체가 이미 사전순으로 정렬된 구조거든요.

사전 크기와 무관한 속도

트라이의 모든 연산은 O(L)입니다(L = 다루는 문자열의 길이). 사전에 단어가 몇 개든 상관없어요.

insert / search / startsWith = O(L)   (단어 수 N과 무관!)
왜 그런가 — 글자 하나마다 배열 인덱스 한 번으로 자식을 찾습니다(child[s[i]-'a']). 비교도, 탐색도 없어요. 그냥 바로 접근이죠. 그래서 글자 수만큼만 시간이 듭니다. 이진 탐색의 log N조차 없어요.
연산트라이정렬+이진탐색해시셋
단어 검색O(L)O(L log N)O(L)
접두사 존재O(L)O(L log N)불가
자동완성O(L+k)O(L log N + k)불가
사전순 출력O(전체) DFS이미 정렬됨불가

⚠️ 공짜는 없다 — 메모리 — 노드마다 child[26] 포인터 배열을 두면, 대부분 NULL인데도 공간을 먹습니다. 노드 10개짜리 우리 예제도 포인터 260개를 갖고 있어요. 그래서 실전에선 맵(map)이나 연결 리스트로 자식을 관리하거나, 압축 트라이(Radix Tree)를 씁니다.

💡 다음 여정 — 아호–코라식 — 트라이에 실패 링크(KMP의 그것!)를 달면, 긴 문장에서 여러 패턴을 한 번에 찾는 아호–코라식이 됩니다. 트라이는 그 자체로도 강력하지만, 문자열 알고리즘의 뼈대이기도 해요. 지금 배운 구조가 다음 편의 기반이 됩니다.
