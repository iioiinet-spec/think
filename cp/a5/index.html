최소 비용 최대 유량 MCMF

가장 싼 증가 경로부터 흘려, 최대 유량을 최소 비용으로 — 역방향 간선은 음수 비용

C 시각화: 용량·비용 · SPFA 최단비용 경로 · 음수 역방향 · 누적 비용

유량에 값을 매기다 — 비용의 등장

에드몬드–카프에서 최대 유량을 배웠습니다. 그런데 현실에서는 어떻게 흘리느냐도 중요해요. 파이프마다 단위당 비용이 다르다면, 같은 양을 흘려도 총비용이 달라집니다.

MCMF(Min Cost Max Flow)는 두 목표를 순서대로 달성합니다 — 먼저 유량을 최대로, 그중에서 비용을 최소로.

용어의미
용량 cap(u,v)간선 u→v로 흘릴 수 있는 최대량
비용 cost(u,v)u→v로 1만큼 흘릴 때 드는 비용
경로 비용경로상 비용의 합 (단위당)
총비용Σ (간선 유량 × 간선 비용)
💡 핵심 반복 — 에드몬드–카프와 뼈대가 같습니다. 다만 증가 경로를 찾을 때 ① BFS(간선 수 최소)가 아니라 ② 비용이 가장 싼 경로를 찾아요. "싼 것부터 쓴다"는 탐욕이 놀랍게도 최적을 보장합니다.

딱 두 가지가 바뀝니다 — ① 경로를 BFS 대신 SPFA로 찾고, ② 역방향 간선에 음수 비용을 답니다. 나머지(잔여 그래프, 병목, 증가 경로 반복)는 에드몬드–카프와 똑같아요. 그런데 왜 다익스트라가 아니라 SPFA일까요? 다음 탭에서 봅니다.

역방향 간선의 비용은 음수다

에드몬드–카프에서 유량을 흘리면 역방향 잔여 간선이 생겼습니다. MCMF에서는 여기에 비용도 붙어요. 그런데 그 비용이 음수입니다.

cost[u][v] = w   →   cost[v][u] = −w
왜 음수인가 — 역방향 간선을 지나는 건 "앞서 u→v로 보낸 유량을 취소"하는 것입니다. 보낼 때 w를 냈으니, 취소하면 w를 돌려받아야 맞죠. 그래서 비용이 −w예요. 회계가 정확히 맞아떨어집니다.
간선 추가 시
/* u→v 용량 c, 비용 w */
cap[u][v] += c;
cost[u][v] = w;
cost[v][u] = -w;   /* 역방향! */

/* cap[v][u]는 0에서 시작 */
/* 유량을 흘려야 >0이 됨 */
회계의 정합성
// u→v로 1 흘림: 비용 +w
// 나중에 v→u로 1 흘림(취소)
//                : 비용 -w

// 합계 = 0
// = 아무것도 안 보낸 것과 같음

// 정확히 원상복구!

그래서 다익스트라를 못 쓴다

음수 비용 간선이 생기니 다익스트라를 쓸 수 없습니다. 다익스트라는 "한 번 확정한 최단거리는 절대 안 줄어든다"를 가정하는데, 음수 간선이 있으면 그 가정이 깨지거든요.

알고리즘음수 간선복잡도MCMF에서
BFS가중치 무시O(E)비용 최소를 못 찾음
다익스트라불가O(E log V)역방향 음수 때문에 ✗
SPFA (벨만–포드)가능평균 O(E)표준 선택 ✓
💡 이 시리즈의 재회 — 벨만–포드를 배울 때 "음수 간선이 있으면 다익스트라 대신 벨만–포드"라고 했습니다. 그 상황이 바로 여기예요! MCMF의 잔여 그래프에는 음수 간선이 있으니 벨만–포드 계열(SPFA)이 필요합니다. 배운 것들이 이렇게 맞물립니다.
SPFA란 — Shortest Path Faster Algorithm. 벨만–포드를 큐로 개선한 것입니다. 모든 간선을 V번 훑는 대신, "거리가 갱신된 정점"만 큐에 넣어 다시 처리해요. 최악은 벨만–포드와 같지만 실전에선 훨씬 빠릅니다.

SPFA 코드

int spfa(void){
    for (int i=1;i<=n;i++){ dist[i]=INF; par[i]=-1; inq[i]=0; }
    dist[S]=0; push(S); inq[S]=1;
    while (큐가 비지 않음){
        int u = pop(); inq[u]=0;
        for (int v=1;v<=n;v++){
            if (cap[u][v] > 0 && dist[u]+cost[u][v] < dist[v]){
                dist[v] = dist[u] + cost[u][v];   /* 완화 */
                par[v] = u;
                if (!inq[v]){ push(v); inq[v]=1; }  /* 큐에 없으면만 */
            }
        }
    }
    return dist[T] < INF;   /* 증가 경로 존재? */
}
💡 잔여 용량이 있는 간선만 봅니다(cap[u][v] > 0). 이게 "잔여 그래프 위에서 최단 비용 경로 찾기"예요. inq[]는 같은 정점이 큐에 중복해 들어가는 걸 막습니다.

마지막 경로의 해부 — 1→3→2→4→6

네 번째 경로가 이 강의의 백미입니다. 한 조각씩 뜯어봅시다.

비용 계산
// 1→3 : 비용 +3  (원본 간선)
// 3→2 : 비용 -1  ← 역방향!
//        (2→3 비용 1의 취소)
// 2→4 : 비용 +4  (원본 간선)
// 4→6 : 비용 +1  (원본 간선)

// 합계: 3 - 1 + 4 + 1 = 7
실제로 일어난 일
// 경로1에서 2→3으로 1 흘렸었음

// 이제 3→2 역방향을 지나며
//   그 유량을 취소
// → 2→3의 최종 유량 = 0!

// 대신 1→3이 직접 3을 담당하고
//      2는 2→4로 빠짐

// = 유량의 재배치
최종 유량 배치를 보면 — 완전 탐색으로 확인한 최적해는 1→2:2, 1→3:2, 2→3:0, 2→4:2, 3→5:2, 4→6:2, 5→6:2입니다. 2→3의 유량이 0이죠! 경로 1이 썼던 그 간선을, 알고리즘이 마지막에 완전히 "마음을 바꿔" 취소했습니다. 역방향 간선이 그 되돌리기를 가능하게 한 거예요.
💡 탐욕인데 왜 최적인가 — MCMF는 매 단계 "지금 가장 싼 경로"만 봅니다(탐욕). 근시안적으로 보이지만, 역방향 간선이 과거의 선택을 되돌릴 여지를 남기기 때문에 최종적으로 최적에 도달해요. 탐욕 + 되돌리기 = 최적. 이게 플로우 계열 알고리즘의 공통 철학입니다.

비용은 절대 줄어들지 않는다

MCMF의 최적성은 하나의 정리에서 나옵니다.

증가 경로의 비용은 매 반복마다 단조 증가 (절대 감소하지 않음)

우리 예제에서 경로 비용이 4 → 5 → 6 → 7이었죠. 우연이 아닙니다. "가장 싼 경로부터" 쓰기 때문에, 다음 경로는 반드시 같거나 더 비싸요.

왜 최적인가 — 유량 k를 흘린 시점의 비용이 "유량 k짜리 모든 방법 중 최소"임을 귀납적으로 보일 수 있습니다. 더 싼 방법이 있었다면, 그 차이가 잔여 그래프에 음수 사이클로 나타나야 하는데, 항상 최단 경로만 썼으니 음수 사이클이 생길 수 없거든요.
