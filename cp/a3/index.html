C 시각화: 구간 갱신 · lazy 배열 · push down · 지연 질의

구간 갱신의 벽 — 왜 지연 전파인가

앞서 배운 세그먼트 트리는 점 갱신(값 하나 바꾸기)은 O(log n)이었습니다. 그런데 "구간 [l,r]의 모든 값에 v를 더하라"는 요청이 오면?

순진하게 구간의 리프를 하나씩 갱신하면 최악 O(n)입니다. 구간이 크면 느려요. 지연 전파(lazy propagation)는 이 구간 갱신을 O(log n)으로 끌어내립니다.

연산점 갱신 세그트리지연 전파
점 갱신O(log n)O(log n)
구간 질의O(log n)O(log n)
구간 갱신O(n)O(log n)
💡 핵심 아이디어 한 줄 — "지금 당장 다 갱신하지 말고, 나중에 필요할 때 미뤄서 하자." 구간을 덮는 노드에 "빚(lazy)"을 표시만 해두고, 실제로 그 아래로 내려갈 일이 생겼을 때 비로소 자식에게 전달합니다.

게으름의 두 가지 장치

🏷️ lazy 배열

각 노드에 "아직 자식에게 안 내려보낸 갱신값"을 저장. 구간 갱신 시 완전히 덮이는 노드에 표시만 하고 멈춥니다.

⬇️ push down

그 노드 아래로 내려가야 할 때(질의·갱신), 쌓인 lazy를 자식에게 전달하고 자기 lazy는 비웁니다.

빚과 상환의 비유 — lazy는 "갚아야 할 빚"입니다. 구간 갱신은 큰 노드에 빚을 기록만 하고(합은 바로 정확히 반영, 자식 전달은 미룸), 나중에 그 노드 아래를 방문할 일이 생기면 그때 자식에게 빚을 넘깁니다(push down). 덕분에 필요한 곳만 갱신해요.

이 강의의 예제

8칸 배열 [5,8,6,3,2,7,2,6]에 구간 합 세그먼트 트리를 만들고, 구간 갱신을 적용합니다.

시나리오
초기 전체합 [0,7] = 39 구간 갱신 [2,5]에 +3 → 전체합 51, 구간합 [2,5]=30 구간 갱신 [4,7]에 +2 → 전체합 59, 구간합 [3,6]=29
💡 [2,5]에 +3을 하면 node5[2,3]과 node6[4,5]에만 lazy=3이 걸립니다. 리프(개별 원소)는 갱신하지 않는데도 모든 질의가 정확해요. 어떻게 그런지 다음 탭부터 한 단계씩 봅니다.

lazy 배열 — 미뤄둔 갱신의 기록

지연 전파의 심장은 lazy 배열입니다. 세그먼트 트리와 같은 크기로 하나 더 두고, 각 노드마다 "아직 자식에게 전달하지 않은 갱신값"을 저장해요.

두 가지가 동시에 유지되는 불변식 — ① tree[node]는 항상 그 노드 구간의 정확한 합입니다(lazy 반영 포함). ② lazy[node]는 자식들에게 아직 안 내려간 갱신입니다. 즉 노드 자신의 합은 맞지만, 자식은 아직 모를 수 있어요.
구간 갱신 시 (완전 포함 노드)
// 구간 [l,r]에 val 더하기
// 이 노드[s,e]가 완전히 덮이면:

tree[node] += val * (e-s+1);
//   구간 크기만큼 합 즉시 반영

lazy[node] += val;
//   자식 전달은 "빚"으로 표시만

// 여기서 멈춤! 자식으로 안 내려감
왜 합은 바로 맞나
// 구간 [s,e] 전체에 val을 더하면
// 합은 val * (구간 크기)만큼 증가

// 예: [2,3]에 +3
//   원소 2개니까 합 += 3*2 = 6
//   tree: 9 → 15 (정확!)

// 개별 원소(리프)는 안 바꿔도
// 구간 합은 이미 정확함
💡 구간 크기 곱셈이 열쇠입니다. 구간 합에서는 "구간의 모든 원소에 val을 더하면 합이 val×(구간 크기)만큼 는다"가 성립해요. 그래서 자식을 일일이 안 건드려도 노드의 합을 정확히 갱신할 수 있습니다.

push down — 빚을 자식에게 넘기기

그런데 나중에 그 노드 아래로 내려가야 할 때가 옵니다(더 작은 구간을 질의하거나 갱신). 그때 쌓인 lazy를 자식에게 전달해야 해요. 이게 push down입니다.

void push_down(int node,int s,int e){
    if (lazy[node] != 0){
        int mid=(s+e)/2, lc=2*node, rc=2*node+1;
        /* 자식의 합에 반영 (각 자식 구간 크기만큼) */
        tree[lc] += lazy[node]*(mid-s+1);
        tree[rc] += lazy[node]*(e-mid);
        /* 자식에게 lazy 전달 (빚 넘기기) */
        lazy[lc] += lazy[node];
        lazy[rc] += lazy[node];
        lazy[node] = 0;  /* 내 빚은 청산 */
    }
}
왜 "내려갈 때만" 하나 — push down은 꼭 필요할 때만 합니다. 노드 아래로 재귀해 들어가기 직전에 호출해요. 그러면 방문하지 않는 가지에는 lazy가 그대로 남아 불필요한 작업을 안 합니다. "게으르게, 필요할 때만 갚는다"가 O(log n)의 비결이에요.

push down이 필요한 순간

lazy로 갱신을 미뤄뒀으니, 이제 질의나 더 작은 구간 갱신이 그 아래로 내려가려 하면 문제가 생깁니다. 자식은 아직 갱신을 모르거든요. 그래서 내려가기 직전에 push down합니다.

먼저 [2,5]에 +3을 적용한 상태(node5·node6에 lazy=3)에서, [3,6] 질의를 해봅시다. 이 질의는 node5·node6을 쪼개서 들어가야 하므로 push down이 일어나요.

질의에서도 push down

질의 함수도 갱신과 똑같이, 완전 포함이 아니어서 자식으로 내려가야 하면 push down을 먼저 합니다.

C — lazy 질의
ll query(int node,int s,int e,
         int l,int r){
    if (r<s || e<l) return 0;
    if (l<=s && e<=r)
        return tree[node];   /* lazy 반영됨 */
    push_down(node,s,e);     /* 내려가기 전! */
    int mid=(s+e)/2;
    return query(2*node,s,mid,l,r)
         + query(2*node+1,mid+1,e,l,r);
}
왜 완전 포함은 push 안 하나
// 완전 포함이면 tree[node]를
// 바로 반환하고 끝

// tree[node]는 이미 정확한 합
// (lazy 반영 포함)

// 자식으로 안 내려가니
// push down 불필요!

// → 게으름 유지
불변식이 지켜주는 정확성 — "tree[node]는 항상 정확, lazy는 자식이 모르는 빚"이라는 불변식 덕분에, 완전 포함 노드는 그냥 읽으면 되고(정확하니까), 쪼개 들어갈 때만 push down하면 됩니다. 이 규칙만 지키면 어떤 질의·갱신 순서에도 답이 정확해요.

연산방문 노드복잡도

구간 질의각 레벨 최대 4개O(log n)

💡 lazy 덕분에 구간 갱신도 구간 질의처럼 각 레벨에서 상수 개 노드만 방문합니다. 그래서 O(log n)이에요. 지연 전파는 "구간에 무언가를 반복 적용하는" 거의 모든 문제(구간 더하기, 구간 대입, 구간 뒤집기 등)에 확장됩니다.

지연 전파 세그먼트 트리 — 전체 코드

구축·구간 갱신·구간 질의·push down을 모두 담은 완성 코드입니다. long long을 쓰는 이유는 구간 갱신이 누적되면 합이 커질 수 있어서예요.

typedef long long ll;
ll tree[4*N], lazy[4*N];

void build(int node,int s,int e){
    if (s==e){ tree[node]=arr[s]; return; }
    int mid=(s+e)/2;
    build(2*node,s,mid); build(2*node+1,mid+1,e);
    tree[node]=tree[2*node]+tree[2*node+1];
}

void push_down(int node,int s,int e){
    if (lazy[node]==0) return;
    int mid=(s+e)/2, lc=2*node, rc=2*node+1;
    tree[lc]+=lazy[node]*(mid-s+1); lazy[lc]+=lazy[node];
    tree[rc]+=lazy[node]*(e-mid);   lazy[rc]+=lazy[node];
    lazy[node]=0;
}

void update(int node,int s,int e,int l,int r,ll val){
    if (r<s || e<l) return;              /* 벗어남 */
    if (l<=s && e<=r){                     /* 완전 포함 */
        tree[node]+=val*(e-s+1);
        lazy[node]+=val; return;           /* 표시만, 멈춤 */
    }
    push_down(node,s,e);                     /* 내려가기 전 */
    int mid=(s+e)/2;
    update(2*node,s,mid,l,r,val);
    update(2*node+1,mid+1,e,l,r,val);
    tree[node]=tree[2*node]+tree[2*node+1];  /* 올라오며 갱신 */
}

ll query(int node,int s,int e,int l,int r){
    if (r<s || e<l) return 0;
    if (l<=s && e<=r) return tree[node];
    push_down(node,s,e);
    int mid=(s+e)/2;
    return query(2*node,s,mid,l,r)+query(2*node+1,mid+1,e,l,r);
}
💡 update와 query의 쌍둥이 구조 — 둘 다 "벗어남 → 완전 포함 → (push down 후) 재귀"의 3경우 패턴입니다. 갱신은 완전 포함에서 lazy를 표시하고, 질의는 완전 포함에서 값을 읽어요. push down은 양쪽 모두 "쪼개 들어가기 직전"에 호출합니다.

확장 — 다른 종류의 lazy

연산 종류lazy가 저장하는 것

구간 대입 + 구간 합덮어쓸 값 + "대입됨" 플래그
구간 더하기 + 구간 최소/최대더할 값 (min/max에 그대로 더함)

일반 원리 — lazy 전파의 뼈대는 항상 같습니다. ① 완전 포함 노드에 연산을 요약해 표시하고, ② 내려갈 때 자식에게 합성해서 전달. 연산이 "구간 전체에 균일하게 적용되고 요약값에 규칙적으로 반영"되기만 하면 지연 전파를 쓸 수 있어요. 대입은 순서가 중요해 플래그가 추가로 필요합니다.
