에드몬드–카프의 낭비 — BFS 한 번에 경로 하나?

에드몬드–카프는 BFS를 한 번 돌려 증가 경로를 하나 찾고, 유량을 흘린 뒤 또 처음부터 BFS를 합니다. 그런데 생각해 보세요 — 같은 길이의 증가 경로가 여러 개라면, BFS 한 번으로 그걸 다 찾아 쓸 수는 없을까요?

디닉의 아이디어 — BFS로 레벨 그래프를 한 번 만들어 놓고, 그 위에서 더 이상 못 흘릴 때까지 여러 경로를 한꺼번에 밀어냅니다(블로킹 플로우). 그러고 나서야 BFS를 다시 해요. BFS 1번 = 경로 여러 개가 되는 겁니다.
에드몬드–카프디닉
BFS 1번당증가 경로 1개증가 경로 여러 개 (블로킹 플로우)

💡 층 그래프의 귀환 — 호프크로프트–카프에서 이분 매칭을 가속할 때 층 그래프 + 다중 증가를 썼죠. 디닉은 바로 그 아이디어를 일반 그래프의 최대 유량으로 가져온 것입니다(역사적으로는 디닉이 먼저!). 같은 발상이 두 문제를 관통해요.

이 강의의 예제 — 에드몬드–카프와 똑같은 네트워크

일부러 에드몬드–카프 편과 완전히 같은 네트워크를 씁니다(CLRS 고전 예제, 6정점 10간선, S=1 T=6). 같은 문제를 두 알고리즘이 어떻게 다르게 푸는지 비교하기 위해서예요.

INPUT (u v c = u→v 용량 c)
6 10 1 6 1 2 16 1 3 13 2 3 10 3 2 4 2 4 12 4 3 9 3 5 14 5 4 7 4 6 20 5 6 4
정답 — 최대 유량 23 (양쪽 동일)
디닉: Phase 1 (레벨 [1:0 2:1 3:1 4:2 5:2 6:3]) 경로 1→2→4→6 12 흘림 경로 1→3→5→6 4 흘림 ← BFS 한 번에 두 경로! Phase 2 (레벨 [1:0 2:1 3:1 4:3 5:2 6:4]) 경로 1→3→5→4→6 7 흘림 최대 유량 = 23, BFS(페이즈) 단 2회 에드몬드–카프는 같은 경로 3개를 BFS 3회로 찾음
💡 같은 경로, 다른 전략 — 놀랍게도 두 알고리즘이 찾는 경로 3개는 완전히 같습니다(12, 4, 7). 차이는 어떻게 묶어서 찾느냐예요. 디닉은 Phase 1에서 12와 4를 한 번의 BFS로 처리합니다. 큰 그래프에서 이 차이가 복잡도를 가릅니다.

디닉의 두 단계

① BFS — 레벨 그래프

S에서의 최단 거리(레벨)를 매깁니다. 그리고 레벨이 정확히 1 오르는 간선만 남긴 그래프를 씁니다.

② DFS — 블로킹 플로우

레벨 그래프에서 S→T 경로가 하나도 안 남을 때까지 DFS로 계속 흘립니다. 이걸 "막힌 유량"이라 해요.

페이즈(phase) — BFS 1번 + 블로킹 플로우 1번을 묶어 한 페이즈라 부릅니다. 페이즈가 끝나면 레벨 그래프를 다시 만들고, T에 도달할 수 없으면 종료해요. 놀라운 사실은 페이즈가 V번을 넘지 않는다는 것입니다(4번 탭에서 증명).
💡 왜 "레벨이 1 오르는 간선만" 쓸까요? 그래야 최단 경로만 따라가고, 빙빙 돌거나 되돌아가지 않습니다. 그 결과 각 페이즈의 DFS가 효율적으로 끝나고, 다음 페이즈에서 최단 거리가 반드시 늘어나요.

레벨 그래프 — 최단 거리로 층을 나누다

BFS로 S에서 각 정점까지의 최단 거리(=레벨)를 잽니다. 잔여 용량이 있는 간선만 따라가요. 그리고 레벨이 정확히 1 오르는 간선만 남깁니다.

간선 u→v를 쓸 수 있다  ⟺  cap[u][v] > 0 그리고 level[v] == level[u] + 1
C — 레벨 그래프 (BFS)
int bfs(void){
    for (int i=1;i<=n;i++) level[i]=-1;
    level[S]=0; push(S);
    while (큐가 비지 않음){
        int u = pop();
        for (int v=1;v<=n;v++)
            if (cap[u][v]>0 && level[v]<0){
                level[v] = level[u]+1;
                push(v);
            }
    }
    return level[T] >= 0;  /* T 도달 가능? */
}
어떤 간선이 빠지나
// Phase 1 레벨:
//   1:0  2:1  3:1  4:2  5:2  6:3

// ✓ 1→2 (0→1)  ✓ 2→4 (1→2)
// ✓ 1→3 (0→1)  ✓ 3→5 (1→2)
// ✓ 4→6 (2→3)  ✓ 5→6 (2→3)

// ✗ 2→3 (1→1) 같은 레벨!
// ✗ 3→2 (1→1) 같은 레벨!
// ✗ 4→3 (2→1) 거꾸로!
// ✗ 5→4 (2→2) 같은 레벨!
💡 10개 간선 중 6개만 레벨 그래프에 남습니다. 같은 레벨끼리 가는 간선(2→3, 3→2, 5→4)과 거꾸로 가는 간선(4→3)이 빠져요. 이렇게 앞으로만 가는 DAG가 되니 DFS가 사이클에 빠지지 않고 효율적입니다.

블로킹 플로우 — 막힐 때까지 민다

블로킹 플로우(blocking flow)란 "레벨 그래프에서 S→T 경로가 하나도 안 남을 때까지 흘린 유량"입니다. 각 페이즈는 블로킹 플로우 하나를 만들고 끝나요.

"블로킹"의 뜻 — 최대 유량이라는 뜻이 아닙니다. 레벨 그래프 안에서 모든 S→T 경로에 포화된 간선이 하나씩은 있다는 뜻이에요. 레벨 그래프 밖(더 긴 경로)으로는 아직 흘릴 수 있고, 그건 다음 페이즈의 몫입니다.
C — DFS (레벨 따라 흘리기)
int dfs(int u, int f){
    if (u==T) return f;      /* 도착! */
    for (int* v=&iter[u]; *v<=n; (*v)++){
        int w = *v;
        if (cap[u][w]>0 && level[w]==level[u]+1){
            int d = dfs(w, min(f, cap[u][w]));
            if (d>0){
                cap[u][w]-=d; cap[w][u]+=d;
                return d;
            }
        }
    }
    return 0;   /* 이 정점에선 T로 못 감 */
}
호출부 — 막힐 때까지
while (bfs()){          /* 페이즈 반복 */
    for (int i=1;i<=n;i++)
        iter[i] = 1;       /* 반복자 초기화 */
    int f;
    while ((f = dfs(S, INF)) > 0)
        flow += f;         /* 계속 흘림! */
    /* f==0 → 블로킹 플로우 완성 */
}
💡 안쪽 while이 핵심입니다. DFS가 0을 반환할 때까지 계속 부르면, 레벨 그래프에서 뽑을 수 있는 경로를 전부 뽑아냅니다. 그게 블로킹 플로우예요.

current-arc 최적화 — 죽은 길은 다시 안 본다

코드에 낯선 게 하나 있죠. iter[u]라는 배열입니다. 이게 디닉을 실제로 빠르게 만드는 열쇠예요.

문제 — DFS를 여러 번 부르는데, 매번 정점 u의 첫 번째 간선부터 다시 훑는다면? 이미 "이 길로는 T에 못 간다"고 판명난 간선을 계속 다시 시도하게 됩니다. 낭비죠.

iter[]의 역할
// iter[u] = u에서 "다음에 볼" 간선

// for (int* v=&iter[u]; ...; (*v)++)
//   ↑ 포인터로 iter[u]를 직접 증가!

// 이 간선이 실패하면 iter[u]++
// → 다음 DFS는 그 다음 간선부터
// → 죽은 길을 두 번 안 봄

// 페이즈 시작마다 iter[] 초기화
효과
// current-arc 없이:
//   한 페이즈가 O(V·E) 넘게 걸릴 수도

// current-arc 있이:
//   각 간선은 페이즈당 O(1)번만 "버려짐"
//   한 페이즈 = O(V·E)

// 페이즈 O(V)회
//   → 전체 O(V²·E)
⚠️ iter[] 초기화 위치가 중요 — 페이즈마다(BFS 직후) 초기화해야 합니다. 레벨 그래프가 새로 만들어졌으니, 이전 페이즈에서 죽었던 간선이 다시 살아날 수 있거든요(우리 예제의 5→4처럼!). 반대로 한 페이즈 안에서는 절대 초기화하면 안 됩니다.

왜 블로킹 플로우가 페이즈를 끝내나

블로킹 플로우가 완성되면, 그 레벨 그래프로는 단 한 방울도 더 못 흘립니다. 그러면 다음 페이즈에서 뭐가 달라질까요?

블로킹 플로우 후 → 다음 페이즈의 level[T]는 반드시 커진다
직관 — 길이 k인 경로가 모두 막혔으니, 이제 T로 가려면 더 긴 길을 찾아야 합니다. 그래서 다음 BFS의 level[T]는 k보다 큽니다. 우리 예제에서 3 → 4로 늘어난 게 그 증거예요. 경로 길이는 최대 V−1이니, 페이즈는 V번 이하입니다.
💡 이게 에드몬드–카프와의 결정적 차이입니다. EK도 "최단 경로 길이가 안 줄어든다"는 성질이 있지만, 같은 길이의 경로마다 BFS를 새로 합니다. 디닉은 같은 길이는 한 페이즈에 몰아서 처리하고 BFS를 아껴요.
